# packaging/

Build scripts and templates for the platform installers. Driven from `.github/workflows/release.yml`; you can also run them locally for testing.

```
packaging/
├── windows/
│   ├── installer.iss                  Inno Setup script → .exe installer
│   └── chocolatey/                    Chocolatey .nupkg (downloads the .exe)
│       ├── gmsoundboard.nuspec
│       └── tools/
│           ├── chocolateyinstall.ps1
│           ├── chocolateyuninstall.ps1
│           └── chocolateybeforemodify.ps1
├── macos/
│   ├── Info.plist.template            Bundle metadata (URL scheme lives here)
│   ├── build-app.sh                   publish/ → .app bundle
│   ├── build-dmg.sh                   .app → .dmg with drag-to-Applications
│   └── homebrew/
│       └── gmsoundboard.rb            Homebrew Cask (lives in a tap repo)
└── linux/
    ├── gmsoundboard.desktop           Launcher entry + MIME registration
    ├── build-appimage.sh              publish/ → .AppImage (portable)
    ├── build-deb-rpm.sh               publish/ → .deb + .rpm (via fpm)
    ├── postinst.sh                    Refreshes desktop DB, registers URI
    └── postrm.sh                      Cleanup hooks
```

## Local testing

Before pushing a release tag, you can sanity-check each builder locally.

**Windows installer** (needs Inno Setup 6 installed):
```powershell
dotnet publish SoundBoard.Desktop -r win-x64 --self-contained `
  -p:PublishSingleFile=true -o publish/win-x64
& "${env:ProgramFiles(x86)}\Inno Setup 6\ISCC.exe" `
  /DAppVersion=0.0.0-dev `
  /DSourceDir=..\..\publish\win-x64 `
  /DOutputDir=..\..\artifacts `
  packaging\windows\installer.iss
```

**macOS .app + .dmg** (on a Mac):
```bash
dotnet publish SoundBoard.Desktop -r osx-arm64 --self-contained \
  -p:PublishSingleFile=true -o publish/osx-arm64
chmod +x packaging/macos/*.sh
packaging/macos/build-app.sh 0.0.0-dev publish/osx-arm64 artifacts
packaging/macos/build-dmg.sh 0.0.0-dev osx-arm64 artifacts
```

**Linux** (on Ubuntu/Debian; needs `appimagetool` + `fpm`):
```bash
sudo gem install --no-document fpm
sudo apt-get install -y rpm
wget https://github.com/AppImage/AppImageKit/releases/download/continuous/appimagetool-x86_64.AppImage \
  -O /usr/local/bin/appimagetool && sudo chmod +x /usr/local/bin/appimagetool

dotnet publish SoundBoard.Desktop -r linux-x64 --self-contained \
  -p:PublishSingleFile=true -o publish/linux-x64
chmod +x packaging/linux/*.sh
packaging/linux/build-appimage.sh 0.0.0-dev publish/linux-x64 artifacts
packaging/linux/build-deb-rpm.sh  0.0.0-dev publish/linux-x64 artifacts
```

## Icons

Each platform reads its icon from a different file format. All three are derived from a single source — `SoundBoard.UI/Assets/app-icon.ico` — and committed to the repo so CI builds don't need image-conversion tooling:

| File | Used by | Generated by |
|---|---|---|
| `SoundBoard.UI/Assets/app-icon.ico` | Windows installer (`SetupIconFile`), Avalonia `Window.Icon`, Chocolatey nuspec `iconUrl` | **Source — edit this one** |
| `packaging/linux/gmsoundboard.png` (256×256) | AppImage, `.deb`, `.rpm` | `generate-icons.py` |
| `packaging/macos/app-icon.icns` (multi-resolution) | macOS `.app` bundle | `generate-icons.py` |

To regenerate the derived files after changing `app-icon.ico`:

```powershell
py -m pip install --user Pillow      # first time only
py packaging/generate-icons.py
```

The script pads the (non-square) source to 256×256, then resamples to PNG and a 7-size ICNS (16/32/64/128/256/512/1024). Commit the output files.

If you ever delete the derived icons, the build scripts have fallbacks: `build-appimage.sh` writes a 1×1 transparent placeholder PNG, and `build-app.sh` builds the `.app` without an icon (macOS shows the generic gear). Releases will still succeed; they'll just look unbranded.

## URI scheme registration — how it lands on each platform

The `gmsound://` URL scheme needs to be registered with the OS so a click on a `gmsound://...` link routes to the app. Each platform handles it differently:

| Platform | Where it's declared | When it activates |
|---|---|---|
| Windows installer | Inno Setup `[Registry]` block writes `HKCU\Software\Classes\gmsound` | At install time |
| Windows portable | App writes the same keys on first launch via `UriSchemeRegistrar` | On first run |
| macOS | `CFBundleURLTypes` in the bundle's `Info.plist` | When the `.app` is moved to `/Applications` |
| Linux .deb / .rpm | `MimeType=x-scheme-handler/gmsound;` in `gmsoundboard.desktop`, plus `xdg-mime default …` in the post-install hook | At install time |
| Linux AppImage | Declared but **not** auto-registered (no install step) | User has to install the `.desktop` file manually, or run [`appimaged`](https://github.com/AppImage/appimaged) |

## Code signing (planned)

We're applying to [SignPath](https://signpath.org/) for free OSS code signing certificates. When approved, two things will happen:

- **Windows**: the Inno Setup installer (and the `.exe` inside it) get authenticode-signed via SignPath's GitHub Action. Uncomment the `SignTool=signpath_signtool` line in `installer.iss` and add the action step to the release workflow.
- **macOS**: SignPath doesn't sign macOS binaries (Apple requires a Developer ID). Without an Apple Developer account ($99/year), the `.app` stays unsigned and users do the right-click→Open workaround on first launch.

The release workflow is written so signing is an additive step — adding it later won't require restructuring the build.
